# AI flow — аналіз мінусів/плюсів оголошень

Короткий огляд того, як працює AI-аналіз описів (кнопка **«AI»** у хедері таблиці).
Детальні специфікації — `docs/plans/llm-analysis.md`; технічні модулі/ендпойнти —
`docs/architecture.md`.

## Що це

Майстер, що знаходить у описах оголошень **мінуси** (недоліки) або **плюси** (переваги) за
заданими критеріями й записує їх у колонки `cons`/`pros` таблиці.

- **Ніколи не авто.** Запускається лише вручну (кнопка «AI»); жоден скан / автооновлення / cron
  його не тригерять.
- **Критерії — на рівні пошуку** (`searches.analysis_criteria`, JSON `{cons:[], pros:[]}`).
- **Мінуси/плюси — на рівні оголошення** (`listings.cons`/`pros`, TEXT `• пункт\n• пункт`,
  сумісно з ручним едітом у `ProsConsCell`).
- **PII продавця в аналіз не йде** (тільки `id/title/description/params`).
- `evidence` (дослівний фрагмент-доказ) **у БД не зберігається** — лише верифікує збіг
  (анти-галюцинація) і показується в кроці перевірки.

## Чотири кроки майстра

1. **Критерії.** Обрати режим (мінуси/плюси) та scope (вибрані / вкладка статусу / весь пошук).
   Критерії можна згенерувати авто (OpenRouter), згенерувати вручну (промпт → чат → вставка)
   або додати власні. Збереження → `searches.analysis_criteria`.
2. **Пошук (matching).** Знайти збіги критеріїв у описах. Два рівноправні рушії (нижче).
3. **Перевірка.** Переглянути знайдені пункти; тегами вмикати/вимикати кожен. Бейдж із
   пунктирною рамкою = `evidence` не підтверджено як підрядок опису.
4. **Запис.** Записати в таблицю. Режим: **«Додати до наявних»** (дефолт — мерж без дублів,
   нічого не затирається) або **«Перезаписати»** (з попередженням, якщо поле вже заповнене).
   Можна повторно прогнати майстер з іншими критеріями й **дописати** нові пункти.

## Два рушії пошуку (крок 2)

### Авто (OpenRouter)
Доступний, якщо в `server/.env` є `OPENROUTER_API_KEY`. Дрібні батчі (по 12), сервер сам
викликає LLM, парсить і верифікує `evidence`. Дефолтна модель —
`google/gemini-2.5-flash-lite`.

### Ручний (ZIP-пакет)
Без ключа. Кнопка «Завантажити ZIP-пакет» віддає архів:
- `prompt.txt` — інструкція з двома варіантами виконання;
- `analyze.py` — **готовий детермінований Python-движок** (regex-матчинг);
- `patterns.example.json` — приклад мапи критеріїв;
- `descriptions/chunk-NNN.json` — оголошення (по 50).

Промпт — **жорстко обмежена однопрохідна процедура** для агента з виконанням коду (Antigravity
CLI, Claude Code), а НЕ дослідницьке завдання. Дозволено рівно 2 кроки:
1. створити ЄДИНИЙ файл `patterns.json` (мапа `критерій → Python-regex`: UA+RU синоніми, сленг,
   морфологічні стеми) — **лише з мовних знань, без інспекції датасету**;
2. запустити `python analyze.py` **рівно один раз** → `output.json` → STOP.

Движок масово сканує всі чанки, **відкидає заперечені збіги в межах клаузи** («екран без
подряпин» → не зараховує), дістає дослівний `evidence`, не друкує нічого в консоль. Так LLM
робить лінгвістику **один раз**, а швидкий детермінований скрипт — масовий прохід.

Промпт містить явний список **ЗАБОРОНЕНОГО** (не створювати інші файли/скрипти, не сканувати
дані, не «вдосконалювати» патерни, не вичищати false positives, не повторювати запуск, не
ітерувати) + умову STOP. Це свідомо прибирає латитуду, через яку слабкі моделі (напр. Gemini
Flash) зациклювалися на нескінченному дата-дослідженні замість видачі результату.

**Fallback** (тільки якщо асистент не вміє запускати код): опрацювати чанки напряму й повернути
той самий JSON — без помічних скриптів.

Результат (`output.json` або відповідь чату) вставляється назад у крок 2 → крок 3 (перевірка)
→ крок 4 (запис).

## Інваріанти

- Зміна `title`/`description` після аналізу → `analysis_stale=1` (бейдж «застарілий аналіз»),
  без авто-переаналізу.
- Промпти — єдине джерело `server/src/analysis/prompts.ts` (спільне для авто й ручного).
- Ключ OpenRouter — лише в `server/.env`, ніколи в код/git.
