# План: зрозумілий звіт аналізу + ширший збір оголошень

## Контекст

Користувач плутається в числах діалогу «Аналіз перед сканом» і не довіряє повноті даних, бо
порівнює їх із «source of truth» на сайті OLX. Розбір виявив **три непорівнювані числа**:

1. **«N оголошень» у звіті** — проста **сума** `visible_total_count` по всіх варіантах запиту
   (`analyzeScan`, `scanner.ts`: `totalListings += plan.rootCount`). Дедупу між синонімами тут
   НЕМАЄ навмисно — одне оголошення під «біговел» і «беговел» рахується двічі.
2. **Дедуп Є лише під час реального скану** (`upsertListings`, `ON CONFLICT(olx_id)`) — реально
   унікальних у базі значно менше за суму. Це й є розрив «сума в аналізі → менше у скані».
3. **Число на сайті OLX** (напр. фасет категорії 2766) — з іншого ендпойнта
   (`/api/v1/offers/metadata/search-categories/`, `docs/olx-api.md` §2.10), не `visible_total_count`.
   Імовірно сайтовий пошук робить морфологічне/міжмовне розширення, тож одне сайтове число вже
   об'єднує те, що додаток розкладає на синоніми. **Гіпотеза** (живої звірки не робимо).

Цілі: (A) зрозумілий звіт — показувати оцінку УНІКАЛЬНИХ + калібрувати реальними даними;
(B) зібрати більше — ширше покриття варіантів запиту з емпіричною самоперевіркою.

## Файли

- `server/src/scanner.ts` — `analyzeScan`: глобальний дедуп вибірки, `estimatedUnique`,
  `sampleUnique` по варіантах, SQL останнього скану (`lastScanRaw`/`lastScanUnique`).
- `server/src/types.ts` + `web/src/types/index.ts` — нові поля `ScanPlan`/`ScanPlanQuery`.
- `web/src/components/searches/action-panel/ScanPlanReportDialog.tsx` — hero, банер,
  калібрування, внесок варіанта.
- `server/src/analysis/prompts.ts` — `buildSynonymsPrompt`.

## Кроки

### A — Зрозумілий звіт

- [x] A1. `analyzeScan`: глобальний `Set<olxId>` по `sampleItems` усіх варіантів; лічильники
      `sumSampleSize`/`unionSampleSize`; per-variant `sampleUnique` (нові olxId на момент варіанта).
- [x] A1. `overlapRatio = unionSampleSize / sumSampleSize`; `estimatedUnique = round(totalListings × overlapRatio)`.
- [x] A1. SQL: останній завершений нормальний скан (`kind != 'analyze' AND found IS NOT NULL`) →
      `lastScanRaw`/`lastScanUnique`.
- [x] A2. DTO `ScanPlanQuery.sampleUnique`, `ScanPlan.estimatedUnique`/`lastScanRaw`/`lastScanUnique`
      (бек + дзеркало фронт).
- [x] A3. Діалог: hero — `≈ N унікальних (оцінка)`, сума — вторинний чип «з дублями»; банер з
      2 тезами (сума-vs-унікальні + різниця з числом OLX); рядок «Минулий скан: X сирих → Y унікальних»;
      «внесок: ~K унікальних» під спектром кожного синоніма.

### B — Зібрати більше — ВІДКОЧЕНО (гіпотезу спростовано live-перевіркою)

Live-перевірка GraphQL OLX (chrome-devtools MCP + node, 2026-06-22) показала: **недозбору немає.**
Розрив «1103 у звіті vs 2766 на сайті» = **фільтр ціни пошуку** (`api_filters: price 200–800 грн`),
а не стеля API чи фаззі-розширення. Доведено: «беговел» без фільтра = 2962 (= фасет `total_count`),
з фільтром 200–800 = 1104 (≈ звітні 1103); «біговел» без фільтра = 3715, з фільтром = 1221 (≈ 1222).
Середовище застосунку (node, без кукі, та сама машина) дає ту саму повну видачу — кукі/регіон ні до чого.

- [x] B1. ВІДКОЧЕНО — `buildSynonymsPrompt` повернуто до початкового стану (розширення било по
      неіснуючій проблемі).

## Test-cases (UI тестує користувач сам)

1. `npm run dev` → пошук із синонімами → «Аналіз перед сканом»: hero `≈ N унікальних (оцінка)`,
   сума — вторинна з міткою «з дублями».
2. Банер пояснює суму-vs-унікальні та різницю з числом OLX.
3. Якщо був скан — рядок «Минулий скан: X сирих → Y унікальних».
4. Під кожним синонімом — «внесок: ~K унікальних».
5. «Згенерувати синоніми» → ширший список (форми/транслітерація/одруки).
6. Повний скан → `found` (унікальні) близько до `estimatedUnique` і нижче за суму.

## Дисклеймер

Причина розриву «сайт vs API» — гіпотеза. Зміни безпечні незалежно: звіт лише прояснює/вимірює,
ширші синоніми тільки збільшують union унікальних.
