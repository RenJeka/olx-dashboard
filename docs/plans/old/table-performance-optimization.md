# План: оптимізація пам'яті та продуктивності таблиці оголошень

## Контекст

Користувач поскаржився, що вкладка застосунку (`http://localhost:5173/`) у Chrome
доростає до **~1 ГБ** RAM і помітно «тормозить». Проведено живу діагностику через
Chrome DevTools MCP (два heap-снапшоти + performance trace, відтворення пагінації,
перемикання пошуків, вводу у фільтр) на реальних даних (пошук «ipad 9» — 1245
оголошень, 1.6 МБ JSON).

**Висновки діагностики:**

- Витоку як такого майже немає: після примусового GC heap падає з ~159 МБ назад до
  ~60 МБ. Проблема — застосунок генерує **величезну кількість сміття на кожну
  взаємодію**, GC не встигає прибирати, а Chrome не повертає системі вже виділену
  пам'ять процесу → footprint росте до 1 ГБ за час роботи вкладки (особливо з
  увімкненим автооновленням, що періодично рефетчить і повністю ре-рендерить таблицю).
- Кожен клік пагінації = **~3 МБ сміття** (24 кліки: 51→118 МБ; цикли туди-сюди до
  159 МБ) при стабільному DOM (~2600 вузлів).
- Повільне справжнє накопичення: **+4 МБ утриманої пам'яті за цикл** — у снапшоті
  ростуть closures клітинок TanStack (`getValue`/`getRenderValue`, 16 185 шт =
  1245 рядків × 13 колонок) і тисячі копій функцій стилізації Chakra
  (`cvaFn`/`cssFn`, 8 679 `LRUCache`).
- Trace: один клік пагінації коштує ~70 мс layout + **~250 мс style recalculation**;
  `patchedFocus` (Chakra) дав **985 мс примусових reflow** за кілька секунд взаємодій.

**Першопричини в коді:**

1. `ListingsTableRow` не мемоїзований (`web/src/components/table/ListingsTableRow.tsx`):
   будь-яка зміна стану таблиці (пагінація, виділення рядка, текст пошуку,
   автооновлення) повністю перерендерює всі 50 рядків × 13 Chakra-клітинок.
2. **Завжди-змонтовані state-машини zag** на кожен рядок: `NoteCell` + `ProsConsCell`×2
   рендерять `Popover.Root` з Portal навіть закритими; колонка «Дата» та
   `DescriptionTooltip` створюють `Tooltip` на кожну клітинку. Разом ~250 машин зі
   слухачами/таймерами на сторінку з 50 рядків — головне джерело сміття й
   `patchedFocus`-reflow.

**Очікуваний результат:** різке скорочення сміття на взаємодію (пагінація, ввід у
фільтр, автооновлення), усунення фризів від style recalc / forced reflow, стабільний
footprint процесу.

**Обсяг (узгоджено з користувачем):** лише «швидкі перемоги» — мемоїзація + лінивий
монтаж важких state-машин + заміна tooltip дати на нативний `title`. Без віртуалізації,
без зміни стеку.

## Файли та зміни (кроки з чекбоксами)

## Що НЕ робимо
- Віртуалізація рядків (`@tanstack/react-virtual`) — відхилено: `pageSize=50` уже
  обмежує DOM, виграш не вартий складності.
- Зміна стеку / схеми / бізнес-логіки. Втручання у backend.

## Перевірка (виконує асистент через Chrome DevTools MCP)

Метрика «до» (заміряно, dev-режим, пошук «ipad 9» 1245 оголошень):
- 24 кліки пагінації: heap 51 → **118 МБ**; цикли вперед-назад до **159 МБ**.
- 8 циклів перемикання пошуків: коливання **64–98 МБ**.
- style recalc ~250 мс/клік, forced reflow `patchedFocus` ~985 мс.

Після реалізації повторити ті самі сценарії й порівняти:
1. `evaluate_script` → `performance.memory.usedJSHeapSize` базово на завантаженій таблиці.
2. 24 кліки пагінації поспіль → приріст heap (ціль: помітно менший за +67 МБ).
3. Цикли вперед-назад → heap не має зростати лінійно так само різко.
4. `performance_start_trace` під час пагінації + вводу → `performance_analyze_insight`
   для `ForcedReflow` і `DOMSize`: час style recalc та reflow має впасти.
5. Функціональна перевірка без регресій (див. чекліст нижче).

## Test-cases (ручна перевірка UI користувачем)

- [ ] **Нотатка:** клік по клітинці відкриває Popover з Textarea; ввід + «Зберегти»
      оновлює текст; повторне відкриття показує збережене.
- [ ] **Плюси/Мінуси:** те саме для обох колонок; зберігається незалежно.
- [ ] **Дата:** наведення показує повну дату+час (нативна підказка); кольорове
      підсвічування свіжих дат збереглося.
- [ ] **Опис:** наведення на клітинку показує підказку з повним текстом (якщо
      увімкнено «розгортання опису»); клік відкриває діалог опису; підсвітка збігів
      працює.
- [ ] **Пагінація:** перехід по сторінках швидкий, без фризів; дані коректні.
- [ ] **Фільтр-пошук:** ввід фільтрує таблицю; підсвітка збігів у назві й описі;
      чекбокси «у назві»/«в описі» діють.
- [ ] **Виділення:** чекбокс рядка/«вибрати всі» виділяє; bulk-панель з'являється;
      зміна пошуку скидає виділення.
- [ ] **Статус:** select-бейдж змінює статус (стає manual).
- [ ] **Сортування / зміна розміру колонок / видимість колонок** — без регресій.
- [ ] **Пам'ять:** за кілька хвилин активної роботи RAM вкладки в диспетчері Chrome
      тримається помітно нижче за 1 ГБ і не росте монотонно.
