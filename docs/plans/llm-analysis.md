## Context

LLM-аналіз описів оголошень: знайти в кожному оголошенні **мінуси** (і окремо — плюси) за
набором **критеріїв**, які задаються/генеруються на рівні пошуку. Реалізується як
**4-етапний майстер** в одному модальному вікні, що відкривається кнопкою **«AI»** у хедері.

Два рівноправні рушії, перемикаються наявністю ключа / вибором користувача:
- **Авто (OpenRouter):** сервер сам шле запити (`google/gemini-2.5-flash-lite`).
- **Ручний (безкоштовний):** для КОЖНОГО LLM-кроку — «Скопіювати промпт» / «Завантажити
  пакет» → користувач проганяє в будь-якому безкоштовному чаті → «Вставити відповідь» →
  сервер парсить. Ключ повністю опціональний — увесь цикл проходиться без нього.

Дзеркальний workflow для плюсів — той самий майстер, режим `pros` (промпти й критерії
інші, механіка ідентична). Перемикач режиму (Мінуси / Плюси) — у першому кроці майстра.

## Модель даних (узгоджено з користувачем 2026-06-12)

- **Критерії — на рівні пошуку.** Нове поле `searches.analysis_criteria` — JSON
  `{ "cons": ["..."], "pros": ["..."] }` (обрані користувачем критерії, окремо для мінусів
  і плюсів). Згенеровані/обрані один раз — діють для всіх оголошень пошуку.
- **Мінуси/плюси — на рівні оголошення.** `listings.cons` / `listings.pros` — масив
  знайдених **criterion** саме для цього оголошення (підмножина критеріїв пошуку). У різних
  оголошень — різні підмножини; порожньо = нічого не знайдено (нормально). Формат у TEXT:
  `• criterion\n• criterion` (сумісно з наявним ручним едітом і майбутнім Excel-експортом).
- **`evidence` (дослівний фрагмент) у БД НЕ зберігається.** LLM повертає `{criterion,
  evidence}`; сервер верифікує `evidence` як substring опису (анти-галюцинація); у превʼю
  (крок 3) показує `criterion` + `evidence`; у `listings.cons`/`pros` пише **тільки масив
  criterion**.

### Зафіксовані рішення

| # | Рішення |
| --- | --- |
| Модель | `google/gemini-2.5-flash-lite` (дефолт, поле в налаштуваннях редаговане) |
| Тригер | Тільки вручну (кнопка «AI» у хедері). Жодного авто зі сканів/автооновлення/cron |
| Обсяг прогону | Перемикач у кроці 1: **вибрані (чекбокси, N)** або **весь пошук (M)** |
| Критерії: генерація | Зважений семпл описів (довгі/змістовні) + **зашитий базовий каркас** (стан, комплектність, «на запчастини/не працює», сліди ремонту, уцінка, торг, відсутність документів); LLM доповнює специфічними. Кнопка «Ще варіанти» (інший семпл). Критерії редаговані (chips: обрати/додати/прибрати) |
| Matching: формат | Нормалізований `criterion` (лексичне середнє: «не вмикається»/«мертвий»/«не реагує» → один критерій) + `evidence` (дослівний фрагмент для верифікації) |
| Анти-галюцинація | Server-side substring-перевірка `evidence` в описі; не пройшло → пункт відкидається |
| Ручний режим | Повний для кожного кроку; видача — **авто-вибір: 1 файл якщо влазить у поріг, інакше кілька** |
| reasoning | Перемикач у налаштуваннях (дефедаулт вимкнено) |
| Чанкування | Авто-режим: дрібні батчі 10–15 (модель деградує на довгому). Ручний: великі батчі (безкоштовні чати мають великий контекст) — менше вставок |
| Перезапис | pros/cons непорожні до аналізу → діалог підтвердження перед перезаписом |
| Мова | Вхід укр/рос; вихід критеріїв і мінусів/плюсів — завжди українською |
| Застарівання | Зміна title/description після аналізу → бейдж «застарілий аналіз», без авто-переаналізу |

## Майстер (UX, 4 кроки в одному модальному вікні)

Кнопка **«AI»** (хедер) → `DialogRoot size="xl"`. Зверху — степер (Критерії → Пошук →
Перевірка → Вставка) + перемикач **Мінуси / Плюси** + перемикач обсягу **вибрані (N) /
весь пошук (M)**.

**Крок 1 — Критерії.**
- Поле перегляду/редагування критеріїв (chips: tap = обрати/зняти; input = додати свій).
- Кнопка «Згенерувати критерії» (авто) АБО — якщо ключа нема / користувач хоче безкоштовно —
  «Згенерувати вручну»: відкривається **бічна панель-помічник** (праворуч, інший
  `bg` — `bg.subtle`) з кнопкою «Скопіювати промпт для генерації критеріїв» + Textarea
  «Вставити відповідь». Після вставки критерії зʼявляються chips-ами для вибору.
- Кнопка «Ще варіанти» (інший семпл). Лічильник «Обрано K із P».

**Крок 2 — Пошук (matching).**
- Кнопка «Знайти» (авто, з прогрес-баром X/N батчів) АБО бічна панель-помічник:
  «Завантажити пакет»/«Скопіювати промпт» (авто-вибір 1 vs кілька файлів) + «Вставити
  відповідь(і)». Підтримка кількох вставок підряд (кожна = свій файл/батч), лічильник
  «Опрацьовано B із усіх».
- Сервер парсить, верифікує `evidence`, складає `{id → criterion[]}`.

**Крок 3 — Перевірка.**
- Окрема таблиця [Фото · Назва · Опис (з підсвіткою evidence) · Знайдені мінуси(критерії)].
- Рядки, де нічого не знайдено — видно окремою позначкою. Відкинуті (невалідний evidence)
  пункти показані закресленими / з іконкою «не підтверджено в описі».
- Експорт цього превʼю: «Excel (.xlsx)» і «JSON» (зберегти для аналізу, навіть якщо
  результат не вставлятимеш) — кнопки тут.

**Крок 4 — Вставка.**
- Кнопки **«Відмінити»** і **«Вставити мінуси у таблицю»**.
- Вставка → chunked PATCH у БД (прогрес-бар), `analysis_at=now` тощо.
- Якщо в когось pros/cons були заповнені раніше → перед вставкою діалог підтвердження
  перезапису (з лічильником M).

  Верифікація evidence — проти `title + description` (критерії можуть бути лише в заголовку).

### Підваріант ручного режиму: скрипт-базований ZIP (агент із виконанням коду)

ZIP-пакет (`/analyze/package.zip`) розрахований на агента, що вміє писати/запускати код
(Antigravity CLI, Claude Code), і містить готовий движок `server/src/analysis/analyze.py`
(кладеться в архів, читається з диску як `schema.sql`) + `patterns.example.json`.

- **ВАРІАНТ А (агент):** агент генерує `patterns.json` (мапа `критерій → Python-regex`:
  UA+RU синоніми, сленг, морфологічні стеми через `\w*`, межі слова; **без інлайн `(?i)`** —
  движок компілює з IGNORECASE) і запускає `python analyze.py`. Движок масово сканує всі
  `descriptions/chunk-*.json`, відкидає заперечені збіги **в межах клаузи** (лівий контекст до
  найближчого `,.;:!?` — всюдисуще «не» з іншої клаузи не глушить критерій), дістає дослівний
  evidence з опису й пише `output.json` (схема як у звичайному matching). **Без stdout** —
  лише файл. Розділення ролей: LLM робить лінгвістику ОДИН раз, детермінований скрипт —
  швидкий масовий прохід (~1–2 с на кілька тисяч оголошень).
- **ВАРІАНТ Б (звичайний чат):** опрацювати чанки напряму й повернути той самий JSON (як було).
- Промпт `buildManualZipInstructions` містить обидва варіанти з чітким розгалуженням.
- [x] `POST /api/listings/analyze/commit` body `{mode, items:[{id, criteria:[]}], model?,
  source:'api'|'import', merge?:'append'|'replace'}` — запис у БД chunked: `pros`/`cons` ←
  `'• '+join`, `analysis_at=now`, `analysis_source`, `analysis_model`, `analysis_stale=0`.
  `merge='append'` (дефолт UI) — додати нові пункти до наявних (парс `• `-рядків, дедуп за
  нормалізованим текстом, нічого не затирається); `merge='replace'` — перезаписати поле (з
  попередженням у UI, якщо поле вже заповнене). Прогрес — клієнтське чанкування (без поллінгу).
